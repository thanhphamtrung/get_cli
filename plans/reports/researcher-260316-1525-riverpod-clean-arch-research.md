# Flutter Clean Architecture with Riverpod - Production Patterns Research
**Date:** 2026-03-16 | **Report:** Comprehensive Research

---

## Executive Summary

Riverpod 3.0 (Sept 2025) represents mature state management for Flutter with strong code generation, clean architecture alignment, and modern async patterns. Key shift: StateNotifier → AsyncNotifier/Notifier. Code generation via `@riverpod` is now standard. Production teams combine Riverpod + Freezed + fpdart + Dio + GoRouter for enterprise-grade apps.

## 1. Riverpod Latest + Code Generation

### Current State (Riverpod 3.0, Sept 2025)

**Recommended Pattern: `@riverpod` Annotations (Code-Generated)**
- Modern standard is *annotation-based code generation* via `riverpod_generator` package
- Replaces manual Provider declarations with cleaner syntax
- Generates `@riverpod` providers from annotated functions/classes
- Riverpod 3.0 supports generic type parameters: `@riverpod T multiply<T extends num>(T a, T b) { return a * b; }`

### How `riverpod_generator` Works

```
@riverpod                      ← Annotation marks provider source
String fetchUser() { ... }     ← Plain function

↓ build_runner processes ↓

Generates:
final fetchUserProvider = FutureProvider.autoDispose<String>((ref) { ... })
```

**Key Benefits:**
- Less boilerplate (no manual provider classes)
- Strong typing enforced
- Cleaner provider naming (auto-generated)
- `autoDispose` applied by default
- Works with async functions via AsyncNotifier

### Setup Requirements
```yaml
dependencies:
  riverpod: ^2.4.0+
  flutter_riverpod: ^2.4.0+
  riverpod_annotation: ^2.1.0+

dev_dependencies:
  build_runner: ^2.4.0+
  riverpod_generator: ^2.3.0+
```

**Run code generation:**
```bash
flutter pub run build_runner build --delete-conflicting-outputs
```

### When to Use Manual vs Code Generation

| Pattern | Use Case | Tradeoff |
|---------|----------|---------|
| `@riverpod` annotations | Default for new code | Requires build_runner |
| Manual Provider() | Legacy code, CI constraints | More boilerplate |

**Current consensus:** @riverpod code generation is now the only recommended path forward in 2025+.

## 2. Clean Architecture + Riverpod Integration

### Layer Mapping

**Clean Architecture (Reso Coder Style)**
```
Presentation Layer  → Widgets, UI logic, Riverpod providers for UI
    ↓ depends on
Domain Layer        → Entities, use cases, repository interfaces
    ↓ depends on
Data Layer          → Repository implementations, datasources, DTOs
```

**Riverpod Provider Hierarchy**

```
presentation/providers/
├── auth_notifier.dart         ← AsyncNotifier managing auth state
├── product_notifier.dart      ← AsyncNotifier for product mutations
└── selectors/
    └── product_selectors.dart ← ref.select() for fine-grained updates

domain/
├── entities/
│   └── product.dart           ← Immutable domain model
├── repositories/
│   └── product_repository.dart ← Abstract interface
└── usecases/
    └── get_products_usecase.dart

data/
├── datasources/
│   └── product_remote_datasource.dart  ← Dio HTTP client
├── models/
│   └── product_dto.dart               ← JSON serialization
└── repositories/
    └── product_repository_impl.dart   ← Repository concrete impl
```

### Provider Patterns by Layer

#### **Data Layer: Repository Provider**
```dart
// lib/data/repositories/product_repository.dart
import 'package:riverpod_annotation/riverpod_annotation.dart';

part 'product_repository.g.dart';

@riverpod
ProductRepository productRepository(ref) {
  final dio = ref.watch(dioClientProvider);
  return ProductRepositoryImpl(dio);
}
```

#### **Domain Layer: UseCase Provider**
```dart
// lib/domain/usecases/get_products_usecase.dart
@riverpod
GetProductsUseCase getProductsUseCase(ref) {
  final repository = ref.watch(productRepositoryProvider);
  return GetProductsUseCase(repository);
}
```

#### **Presentation Layer: AsyncNotifier**
```dart
// lib/presentation/providers/products_notifier.dart
part 'products_notifier.g.dart';

@riverpod
class ProductsNotifier extends _$ProductsNotifier {
  @override
  Future<List<Product>> build() async {
    final usecase = ref.watch(getProductsUseCaseProvider);
    return usecase.call();  // Auto-wrapped in AsyncValue<List<Product>>
  }

  // Public method for UI mutations
  Future<void> refresh() => ref.refresh(productsFutureProvider);
}

// Provider generated by @riverpod
final productsProvider = ProductsNotifier().future;  // FutureProvider
```

### State Management Patterns

| Pattern | Use Case | Tradeoff |
|---------|----------|---------|
| `@riverpod` FutureProvider | Read-only async data (API calls) | Auto AsyncValue wrapping |
| `@riverpod class` AsyncNotifier | Read + Write async (mutations) | More boilerplate, full control |
| `@riverpod` Notifier | Sync state with mutations | Avoid; prefer AsyncNotifier |
| StateNotifier (legacy) | Old code | ❌ Deprecated in Riverpod 3.0 |

**Modern production pattern (2025):**
```dart
// Async reads: Use @riverpod FutureProvider (code-generated)
@riverpod
Future<User> fetchUser(ref, String id) async { ... }

// Async reads + writes: Use @riverpod class AsyncNotifier
@riverpod
class UserNotifier extends _$UserNotifier {
  @override
  Future<User> build() async { ... }

  Future<void> update(User user) async { ... }
}
```

### Dependency Flow Example

```dart
// User taps "Refresh Products" button
ElevatedButton(
  onPressed: () => ref.refresh(productsProvider),
  child: Text('Refresh'),
)

// Riverpod graph:
productsProvider (AsyncNotifier)
  ↓ calls usecase
getProductsUseCaseProvider
  ↓ depends on repo
productRepositoryProvider
  ↓ depends on HTTP client
dioClientProvider (Dio instance)
```

## 3. Dio + Riverpod Integration

### Best Practice Setup

**HTTP Client Provider (Singleton)**
```dart
// lib/data/providers/dio_provider.dart
import 'package:dio/dio.dart';
import 'package:riverpod_annotation/riverpod_annotation.dart';

part 'dio_provider.g.dart';

@riverpod
Dio dioClient(ref) {
  final dio = Dio(
    BaseOptions(
      baseUrl: 'https://api.example.com',
      connectTimeout: Duration(seconds: 30),
      receiveTimeout: Duration(seconds: 30),
      headers: {'Accept': 'application/json'},
    ),
  );

  // Add interceptors
  dio.interceptors.add(LoggingInterceptor());

  // Bind to auth state for token injection
  final auth = ref.watch(authNotifierProvider);
  auth.whenData((user) {
    dio.options.headers['Authorization'] = 'Bearer ${user.token}';
  });

  return dio;
}
```

**Error Handling Strategy**
```dart
// Dio throws on HTTP errors → Riverpod catches → AsyncValue.error
@riverpod
Future<List<Product>> fetchProducts(ref) async {
  final dio = ref.watch(dioClientProvider);
  try {
    final response = await dio.get('/products');
    // Riverpod auto-wraps in AsyncValue<List<Product>>
    return (response.data as List)
        .map((p) => Product.fromJson(p))
        .toList();
  } catch (e) {
    // Riverpod catches and sets AsyncValue.error
    rethrow;
  }
}
```

### OAuth 2.0 with Riverpod + Dio

**Package:** `riverpod_oauth_dio` (oven-ready solution)
- Supports Client Credentials, Password, Refresh grants
- Auto-injects OAuth interceptor into Dio
- Manages token refresh lifecycle

```dart
// Auto-configured OAuth + Dio
@riverpod
Dio dioWithOAuth(ref) {
  return createDioWithOAuth(
    clientId: 'your_client_id',
    clientSecret: 'your_client_secret',
    tokenUrl: 'https://auth.example.com/token',
  );
}
```

### Testing with Dio + Riverpod

**Offline Mocking Pattern**
```dart
// Create StateProvider to toggle offline mode
final mockOfflineProvider = StateProvider<bool>((ref) => false);

// Dio interceptor checks provider state
dio.interceptors.add(
  InterceptorsWrapper(
    onRequest: (options, handler) {
      if (ref.read(mockOfflineProvider)) {
        return handler.reject(DioException(...));
      }
      return handler.next(options);
    },
  ),
);
```

## 4. GoRouter + Riverpod Integration

### Authentication Guard Pattern (2025 Standard)

**Recommended: Listenable + Redirect Function**

```dart
// lib/presentation/routes/app_router.dart
import 'package:go_router/go_router.dart';
import 'package:riverpod_flutter_binding/riverpod_flutter_binding.dart';

final appRouterProvider = Provider((ref) {
  final authState = ref.watch(authNotifierProvider);

  return GoRouter(
    refreshListenable: Listenable.merge([
      // Notify GoRouter when auth state changes
      ref.read(authNotifierProvider.notifier),
    ]),
    redirect: (context, state) {
      // Run auth logic before every route
      if (authState.isLoading) {
        return '/splash';
      }

      if (authState.hasError) {
        return '/login';
      }

      final isLoggedIn = authState.whenData((user) => user != null).value ?? false;
      final isLoginRoute = state.matchedLocation == '/login';

      if (!isLoggedIn && !isLoginRoute) {
        return '/login';
      }

      if (isLoggedIn && isLoginRoute) {
        return '/home';
      }

      return null; // No redirect
    },
    routes: [
      GoRoute(path: '/login', builder: (_, __) => LoginPage()),
      GoRoute(path: '/home', builder: (_, __) => HomePage()),
    ],
  );
});
```

**AuthNotifier Structure**
```dart
@riverpod
class AuthNotifier extends _$AuthNotifier implements Listenable {
  late final _listeners = <VoidCallback>[];

  @override
  Future<User?> build() async {
    // Load token from secure storage
    final token = await SecureStorage.getToken();
    return token != null ? User(token: token) : null;
  }

  Future<void> login(String email, String password) async {
    state = const AsyncValue.loading();
    state = await AsyncValue.guard(() async {
      final user = await _authRepo.login(email, password);
      _notifyListeners();  // Trigger GoRouter refresh
      return user;
    });
  }

  @override
  void addListener(VoidCallback listener) => _listeners.add(listener);

  @override
  void removeListener(VoidCallback listener) => _listeners.remove(listener);

  void _notifyListeners() {
    for (final listener in _listeners) listener();
  }
}
```

### Per-Route Guard Pattern (Alternative)

```dart
GoRoute(
  path: '/admin',
  builder: (_, state) => AdminPage(),
  redirect: (context, state) {
    final authState = ref.read(authNotifierProvider);
    final isAdmin = authState.whenData((user) => user?.role == 'admin').value ?? false;

    return !isAdmin ? '/unauthorized' : null;
  },
)
```

## 6. fpdart Either Pattern

### Current Usage Pattern (2025)

**Key Finding:** Either<L, R> pattern is *declining* in Riverpod-heavy codebases.

**Why:** Riverpod's `AsyncValue<T>` already handles error states. Using Either adds redundant error matching.

```dart
// ❌ Discouraged (2025): Double error handling
Either<Failure, List<Product>> getProducts();

// ✅ Recommended: Single error layer via AsyncValue
Future<List<Product>> getProducts();  // Riverpod wraps in AsyncValue
```

### When Either Still Makes Sense

**Use Either for domain-layer business logic** (non-Riverpod):

```dart
// lib/domain/usecases/get_products_usecase.dart
class GetProductsUseCase {
  Future<Either<Failure, List<Product>>> call() async {
    try {
      final products = await _repository.getProducts();
      if (products.isEmpty) {
        return Left(NoProductsFailure());
      }
      return Right(products);
    } catch (e) {
      return Left(ServerFailure());
    }
  }
}

// Convert to AsyncValue in Riverpod layer
@riverpod
Future<List<Product>> products(ref) async {
  final usecase = ref.watch(getProductsUseCaseProvider);
  final result = await usecase.call();

  return result.fold(
    (failure) => throw Exception(failure.toString()),  // Riverpod catches
    (products) => products,
  );
}
```

### Advanced: ReaderTaskEither

For dependency + async + error in one:
```dart
typedef GetProductsTask = ReaderTaskEither<Dependencies, Failure, List<Product>>;

GetProductsTask getProducts = (dependencies) =>
  TaskEither.tryCatch(
    () => dependencies.api.getProducts(),
    (e, _) => ServerFailure(),
  );
```

**Current consensus:** Flatten to Riverpod's AsyncValue for UI. Keep Either for pure domain logic only.

## 7. Envied Setup (Environment Variables)

### Recommended Setup (2025)

```yaml
dependencies:
  envied: ^0.3.0

dev_dependencies:
  envied_generator: ^0.3.0
  build_runner: ^2.4.0
```

### Configuration

**Create `.env` file** (add to `.gitignore`):
```
API_BASE_URL=https://api.example.com
API_KEY=secret_key_123
```

**Create `lib/core/config/env.dart`**:
```dart
import 'package:envied/envied.dart';

part 'env.g.dart';

@Envied(path: '.env', obfuscate: true)
abstract class Env {
  @EnviedField(varName: 'API_BASE_URL')
  static const String apiBaseUrl = _;

  @EnviedField(varName: 'API_KEY')
  static const String apiKey = _;
}
```

**Run code generation:**
```bash
flutter pub run build_runner build --delete-conflicting-outputs
```

### Integration with Riverpod

```dart
// lib/data/providers/dio_provider.dart
@riverpod
Dio dioClient(ref) {
  return Dio(
    BaseOptions(
      baseUrl: Env.apiBaseUrl,  // From generated env
      headers: {'X-API-Key': Env.apiKey},
    ),
  );
}
```

### Obfuscation

- `obfuscate: true` embeds values in code (harder to reverse-engineer)
- Still add `.env` to `.gitignore`
- Safe for committed app bundles

## 9. Riverpod Modifiers & DI Patterns

### `.family` Modifier (Parameterized Providers)

**Use Case:** Different state per parameter (user ID, query, page)

```dart
// lib/presentation/providers/user_provider.dart
@riverpod
Future<User> fetchUser(ref, String userId) async {
  // Generates unique provider per userId
  // Caches separately per parameter combination
  final api = ref.watch(dioClientProvider);
  return api.getUser(userId);
}

// Watch in widget
@override
Widget build(BuildContext context, WidgetRef ref) {
  final user = ref.watch(fetchUserProvider('user_123'));
  return user.when(...);
}
```

**Key:** Parameters must have consistent `==` and `hashCode`. Avoid passing objects with mutable state.

### `.autoDispose` Modifier

**Use Case:** Cleanup unused providers automatically

```dart
@riverpod  // autoDispose is default with code generation
Future<List<Product>> products(ref) async { ... }

// Without code generation (manual):
final productsProvider = FutureProvider.autoDispose<List<Product>>((ref) async { ... });
```

**Benefits:**
- Memory cleanup when widget unmounts
- Prevents cache bloat
- Reduces listener count

### `.select()` for Performance

**Reduce rebuilds by watching specific fields:**

```dart
// Only rebuild when user.name changes, ignore other fields
final userName = ref.watch(
  fetchUserProvider(userId).select((user) => user.name)
);
```

### `.notifier` Access Pattern

```dart
// Get notifier for mutations without watching state
final notifier = ref.read(authNotifierProvider.notifier);
await notifier.logout();
```

### Scoped Providers (DI per Scope)

```dart
// Override for tests/flavors
final dioProvider = Provider((ref) => Dio());

testWidgets('login', (tester) async {
  await tester.pumpWidget(
    ProviderScope(
      overrides: [
        dioProvider.overrideWithValue(MockDio()),
      ],
      child: const MyApp(),
    ),
  );
});
```

## 10. Riverpod 3.0 Major Changes

### StateNotifier → AsyncNotifier Migration (Mandatory by end 2025)

**What Changed:**
- Riverpod 2.0: StateNotifier + new Notifier/AsyncNotifier coexist
- Riverpod 3.0 (Sept 2025): StateNotifier moved to `legacy.dart`, not removed but discouraged

```dart
// ❌ Old (Riverpod 2.0):
final userProvider = StateNotifierProvider((ref) => UserNotifier());

// ✅ New (Riverpod 3.0):
@riverpod
class UserNotifier extends _$UserNotifier {
  @override
  Future<User> build() async { ... }
}
```

### Generic Type Support

```dart
@riverpod
T multiply<T extends num>(T a, T b) => a * b;

// Called with type parameter
ref.watch(multiplyProvider(10, 20));  // T inferred from args
```

### FamilyNotifier Removal

```dart
// ❌ Old (Riverpod 2.0):
class UserNotifier extends FamilyAsyncNotifier<User, String> { ... }

// ✅ New (Riverpod 3.0):
@riverpod
class UserNotifier extends _$UserNotifier {
  @override
  Future<User> build(String userId) async { ... }
}
```

## 11. Production Patterns Checklist

### Essential Setup
- [x] `@riverpod` code generation (not manual providers)
- [x] AsyncNotifier for state + mutations
- [x] Freezed for immutable models
- [x] Dio with singleton provider
- [x] GoRouter with `Listenable` for auth guards
- [x] Envied for environment variables
- [x] Layer separation (data/domain/presentation)

### Performance Optimization
- [x] Use `ref.select()` to reduce widget rebuilds
- [x] Apply `.autoDispose` for cleanup
- [x] Use `.family` for parameterized caching
- [x] Lazy-load heavy providers with `.lazy()`

### Error Handling
- [x] Let Riverpod's AsyncValue handle errors (not Either)
- [x] Use sealed classes + pattern matching for domain errors
- [x] Catch Dio exceptions → Riverpod AsyncValue.error

### Testing
- [x] Override providers via ProviderScope
- [x] Mock Dio/repositories at data layer
- [x] Test AsyncNotifier state transitions
- [x] Use `golden_toolkit` for widget tests

### Security
- [x] Envied with `obfuscate: true` for sensitive vars
- [x] Store auth tokens in secure storage (not SharedPreferences)
- [x] Validate token refresh in auth interceptor
- [x] NEVER commit `.env` or generated `env.g.dart`
